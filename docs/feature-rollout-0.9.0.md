# 0.9.0 Feature Rollout Defaults

This page lists the shipped 0.9.0 posture and opt-in path for its new surfaces.
The SQL classifier, profile ceiling, confirmation grants, rollback defaults,
and audit requirements remain authoritative underneath every transport feature.

| Surface | Default | Enable or disable | Safety rationale |
|---|---|---|---|
| Dedicated control listener | Off because `[http.control]` is absent. | Configure native `[http.tls]` with `client_ca_path`, register the client leaf fingerprint in `[http.mtls]`, allow the matching `mtls:<fingerprint>` under `[http.operator]`, then add `[http.control]`. Remove `[http.control]` to disable it. | The listener authenticates mTLS identity before HTTP parsing and uses separate bounded pre-auth, operator, and readiness worker pools. It never serves ordinary MCP or dashboard routes. |
| Stateful HTTP notifications | Negotiated only for a stateful HTTP MCP session; queues and replay are scoped to the owning session and outbound stream. | Configure `[http].stateful = true` and use a client that advertises and handles notifications. Disable stateful framing to keep the ordinary response path. | Foreign cursors are refused, progress tokens are isolated, and retention loss produces a deterministic gap instead of cross-session replay. |
| Opaque session leases | Always on for lease-backed operations; handles are generated by the server and bound to the authenticated owner. | Acquire, renew, and release only through the session tools. There is no predictable-ID compatibility mode. | Linearized revocation and dirty-session quarantine prevent a revoked or uncertain Oracle session from re-entering the pool. |
| Non-transactional effect metadata | Always present in the 0.9.0 `GuardDecision` API. | Library callers constructing the struct initialize `non_transactional_effect` and `query_effect_requires_fetch`; operators do not toggle these fields. | Sequence consumption and query-shaped effects remain explicit and confirmation-gated even when ordinary DML would roll back. |
| SQL-free current audit records | On for new 0.9.0 records; historical records are verified byte-for-byte and are not rewritten. | No downgrade toggle. Preserve the historical chain and use the 0.9.0 verifier for its newer segment. | Exact and normalized SQL hashes retain correlation while new signed records replace raw SQL preview text with a fixed redaction marker. |
